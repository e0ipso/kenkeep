Any artifact kenkeep generates or documents — config templates, docs sections, warning boxes, nudge copy, prompts — treats all supported harnesses even-handedly. kenkeep is harness-agnostic and has no default harness, so do not bake one harness's discriminator value or one vendor's model into an example as if it were the default.

Two acceptable patterns. Use neutral placeholders that reference a per-harness shapes table: the `config.yaml` header generated by `defaultProjectConfigBody()` in `src/lib/settings.ts` writes `harness: <harness>` plus a shapes table covering every adapter, not a concrete `harness: claude` stanza. Or, when concrete examples genuinely help, spread them evenly — state the cognitive requirement once (e.g. "mid-tier model at moderate effort"), then give one example per supported harness: Claude → `sonnet`/`medium`, Codex → `gpt-5-codex`/`low`, Cursor/OpenCode → user selects any mid-tier model in their provider.

The rationale: Cursor and OpenCode let users choose any model family, so naming one provider's model leaves those harnesses underspecified, and defaulting an example stanza to `harness: claude` implies a default that does not exist and biases users toward one runtime.
